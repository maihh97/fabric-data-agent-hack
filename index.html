Build a Fabric Data Agent

Today you'll build a conversational data agent that reasons over two datasets. Out of the box the agent gets some answers wrong; your job is to investigate why and configure it until it's reliable. Use the menu on the left to move through the guide.

The challenge data

Two community mental-health datasets that share one patient key - LPI_Pseudonym.

community_data ~254k rows

One row per community appointment.

Directorate, LPI_Pseudonym, AttendanceDate, ContactDuration, AttendedOrDNADesc, FirstAttendanceDesc, TeamAtAppointmentDesc

inpatient_data ~9.5k rows

One row per inpatient spell.

LPI_Pseudonym, Gender, NationalEthnicity, AdmissionDate, DischargeDate, SpellNumber, DischargeDestination, WardName, Directorate, Virtual, WardSpecialtyDescription, ReAdmittedWithin28Days, AdmissionWard, DischargeWard

The problem

Why we're hereRight now, with no configuration, the Fabric Data Agent generates incorrect answers to several of these questions. The data has quirks the agent can't guess on its own - so it miscounts, splits categories, or picks the wrong column. Let's try to improve it.

Questions the bare agent struggles with

QuestionWhat trips the agent up
How many patients were readmitted within 28 days?ReAdmittedWithin28Days is 1 or NULL - there's no 0. NULL means "not readmitted."
How many DNAs in Tower Hamlets last December?"Tower Hamlets" and "TowerHamlets" both appear (~10k each) and dates are DD/MM/YYYY.
Which patients had a community DNA and were later admitted?Needs a join across both sources on LPI_Pseudonym - and the right source routing.
Average contact duration for attended appointments?"Attended" isn't a column - it's AttendedOrDNADesc = 'On time : Seen'.

How to investigate a wrong answer

Before changing anything, learn why the agent answered wrong. Try the following:

  • Run steps view - after any answer, expand it to see the generated SQL/DAX/KQL, which data source was chosen, and which example queries were retrieved.
  • Diagnose checklist - is an agent instruction missing or vague? Is the question ambiguous given the schema? Are values inconsistently formatted?
  • Diagnostics export - download a full snapshot of the agent's configuration and execution steps for deeper troubleshooting.

Judging criteria

Two ways to score: answer the benchmark questions correctly, and try to break everyone else's agent.

Core - answer correctly

Your agent must return the right answer to every benchmark question in the exercise workbook's question tabs. From the community question set:

  • How many appointments were there in December 2025?
  • What proportion of appointments were cancelled by the patient?
  • How many face-to-face appointments were there in December 2025?
  • Which patient had the most appointments in December 2025?

Bonus - break an agent

Think your agent is bulletproof? Pose a question that makes another team's agent answer incorrectly or fail - and you earn bonus points.

This rewards adversarial thinking: edge cases, ambiguous wording, tricky joins, the data quirks above. Every break you find is a gap a real user could hit too.

Section 02

Data Agent Introduction

What a Fabric data agent is, and what happens between your question and the answer.

A Fabric data agent is a conversational Q&A system over your governed data in OneLake. Anyone can ask questions in plain English and get precise, context-rich answers - no need to know the schema or write a query.

How it works

It uses large language models to interpret the question, then runs a five-stage loop:

  1. Understand It interprets your natural-language question.
  2. Check permissions It uses your credentials, respects Microsoft Purview governance, and stays strictly read-only.
  3. Route to a source It picks the most relevant connected source for the question.
  4. Generate & validate It translates the question into SQL, DAX, or KQL and checks it before running.
  5. Execute & respond It runs the query and returns a table, summary, or key insights.

It speaks every query language

You ask in plain English; the agent writes the right query for each source.

Your question becomes…For these sources
SQLLakehouse, Warehouse, SQL / Mirrored databases
DAXPower BI semantic models
KQLKQL / Eventhouse databases

What it connects to

Up to five data sources per agent, in any combination: lakehouse, warehouse, Power BI semantic model, KQL database, mirrored database, ontology, and Microsoft Graph. For this challenge you'll connect community_data and inpatient_data.

Read-only by designThe agent only ever generates "read" queries - it never creates, updates, or deletes your data.

Before you start: prerequisites

  • A paid F2 or higher Fabric capacity.
  • At least one data source containing data: lakehouse, warehouse, Power BI semantic model, KQL database, mirrored database, or ontology.
  • Read access to that source - you can't query what you can't see.

Good to know - limits & quirks

Knowing these up front will save you debugging time during the build:

LimitWhat it means for you
Read-onlyGenerates SQL/DAX/KQL read queries only - never create, update, or delete, and it won't trigger notebooks or jobs.
25 × 25Chat responses are capped at roughly 25 rows and 25 columns and may be summarised - it's built for insights, not full data exports.
English onlyQuestions, instructions, and example queries should be written in English.
Tables, not filesNo unstructured data (PDF/Word/txt). Lakehouse CSV/JSON must be ingested into tables first.
Same regionThe data source's capacity and the agent's capacity must be in the same region, or queries fail.
100 examplesUp to 100 example queries per data source.
Fixed modelYou can't change the underlying LLM. Chat history may occasionally reset after service updates, and prior turns can influence later answers.

Who's in charge: governance & precedence

The agent always runs under the asking user's credentials, and Microsoft Purview policies (DLP, sensitivity labels, access restrictions) apply throughout. When instructions conflict, higher layers always win - no prompt can override a higher layer:

  1. Organisational intent - tenant-wide admin policies & compliance (highest).
  2. Role-based intent - workspace governance and permission boundaries.
  3. Developer intent - the instructions and examples you configure.
  4. User intent - the end user's question (lowest).

Section 03

Configure Your Agent

This is where you add the business context in.

Build it from scratch

Building an agent is like building a Power BI report - design, refine, then publish and share. The setup order:

  1. Select data sources Add community_data and inpatient_data (up to five sources allowed).
  2. Choose tables Select only the relevant tables/columns per source - keep each source to 25 tables or fewer.
  3. Add instructions Write agent-level and data-source instructions (below).
  4. Add example queries Seed few-shot examples (section 04).
  5. Test, then publish Iterate against your failing questions, then publish for the team.

The four configuration levers

1. Agent instructions

Top-level guidance: the agent's objective, which source to prioritise, key terminology, and response style. A starting structure tuned to this challenge:

## Objective
Answer questions about community appointments and inpatient
spells for a NHS trust.

## Data sources
Use 'community_data' for appointments, attendance and DNAs.
Use 'inpatient_data' for admissions, wards and readmissions.

## Key terminology
- "DNA" means Did Not Attend: AttendedOrDNADesc = 'DNA - no advance warning'.
- "Attended" means AttendedOrDNADesc = 'On time : Seen'.
- "Readmission" means ReAdmittedWithin28Days = 1.
- Dates are formatted DD/MM/YYYY.

## Response guidelines
Show counts in a table; state the date range you filtered on.

2. Data source instructions

Applied once the agent routes to a source. Describe tables, key columns, relationships, NULL handling, and typical value formats - the agent can't see row values before it queries.

## inpatient_data - general knowledge
One row per inpatient spell. Join to community_data on LPI_Pseudonym.

## When asked about readmissions
ReAdmittedWithin28Days is 1 (readmitted) or NULL (not readmitted) -
there is no 0. Count readmissions with WHERE ReAdmittedWithin28Days = 1.
Never count NULL as a readmission.

3. Data source description

One focused line per source so the agent can route correctly at a glance.

Examplecommunity_data: "Community mental-health appointments - attendance, DNAs, contact duration and team, by directorate."

4. Provide Examples

Sample question-and-query pairs the agent learns from. Covered in full in section 04 - they're your most direct fix for the failing questions.

Best practices that matter most here

Define terms, synonyms & inconsistent values

Your data has a real inconsistency: Tower Hamlets and TowerHamlets are the same directorate. Tell the agent, or counts split in two.

✗ Bare agent

Counts "Tower Hamlets" only - misses ~10k "TowerHamlets" rows.

✓ With instruction

"Tower Hamlets is recorded as both 'Tower Hamlets' and 'TowerHamlets' - always include both."

Say what to do, not just what not to do

✗ Less effective

"Don't miscount readmissions."

✓ Better

"Count readmissions as ReAdmittedWithin28Days = 1; treat NULL as not readmitted."

Other habits

  • Get data AI-ready - descriptive names beat col1, flag. (Your columns are already well-named - good.)
  • Specialise the agent - keep it focused on this mental-health scenario, not "all NHS questions."
  • Minimise scope - only the tables/columns you need; 25 tables or fewer per source.
  • Use leading words - drop syntax hints like LIKE '%...%' to steer query shape.
  • Keep instructions concise - clear and purposeful beats long and vague.

Section 04

Provide Examples

Few-shot examples give the agent concrete patterns to copy - your highest-leverage fix for wrong answers.

Each example pairs a natural-language question with its query. When a user asks something, the agent retrieves the most relevant examples (typically the top three to four) and feeds them into generation, so it learns the expected structure, filters, and joins.

Worked examples for this challenge

Seed examples that target exactly the questions the bare agent fails:

examples = {
  # Readmissions: filter = 1, never count NULL
  "How many patients were readmitted within 28 days in 2025?":
    "SELECT COUNT(DISTINCT LPI_Pseudonym) AS readmitted "
    "FROM inpatient_data "
    "WHERE ReAdmittedWithin28Days = 1 "        # NULL = not readmitted
    "AND AdmissionDate >= '2025-01-01';",

  # Tower Hamlets recorded two ways; DNA is a specific value
  "How many DNAs in Tower Hamlets in December 2025?":
    "SELECT COUNT(*) AS dnas FROM community_data "
    "WHERE AttendedOrDNADesc = 'DNA - no advance warning' "
    "AND Directorate IN ('Tower Hamlets','TowerHamlets') "
    "AND AttendanceDate >= '2025-12-01' "
    "AND AttendanceDate <  '2026-01-01';",
}
datasource.add_fewshots(examples)

Which sources support example queries

Data sourceExample queries?
Lakehouse✓ Yes
Warehouse✓ Yes
Eventhouse KQL databases✓ Yes
Semantic models✗ No
Ontology✗ No
TipIf you load these CSVs into a lakehouse or warehouse, example queries work. Add comments like -- NULL = not readmitted to flag important logic.

Validate them with the SDK

# Validate your natural-language / SQL pairs
result = datasource.evaluate_few_shots(batch_size=20)
print(f"Success rate: {result.success_rate:.2f}% "
      f"({result.success_count}/{result.total_examples})")
NoteThe validator currently supports SQL-based examples only - not KQL.

The three validator scores

Clarity

Is the question specific? "DNAs in Tower Hamlets in Dec 2025" beats "show attendance."

Relatedness

Does the SQL answer it - right metric, filters, granularity?

Mapping

Does every literal (the date, the directorate) appear in the SQL?

An example is high quality only when all three scores are positive.

Conflict detection

After quality checks, the SDK flags conflicting examples - same intent but different tables or aggregation logic. Resolving them makes the agent more deterministic. Keep each example distinct and non-overlapping.

Section 05

Improve Accuracy

Two loops that turn a flaky agent into a reliable one: get routing right, then iterate on failures.

Data source routing

With two sources, the agent must pick the right one for each question. That choice is routing. Pick wrong, and you get incomplete or empty answers - e.g. a readmission question accidentally sent to community_data.

Signs routing needs work

  • It picks the wrong source for a question you expected a specific source to answer.
  • It says it can't find an answer that clearly exists in a connected source.
  • It gives different answers to similar questions because it keeps switching sources.

Expand the run steps to confirm which source was used. If a routing tool appears as its own step, the decision was ambiguous - a signal to add more signal.

Improve routing - in order

  1. Tighten schema selection Select only the tables and columns each source should cover.
  2. Add a data source description One focused line on what each source contains.
  3. Add example queries Especially for questions that previously routed to the wrong source.
  4. Add explicit routing rules As a last resort, declare them in agent instructions:
## Topics
- Appointments, attendance, DNAs, contact duration → community_data
- Admissions, wards, discharges, readmissions       → inpatient_data
- "Patients who did X and also Y" → query both, join on LPI_Pseudonym
Keep it leanLong rule lists crowd out other instructions and need updating when sources change. Use them sparingly.

Iterate: a loop, not a one-time setup

Tuning is experimentation, observation, and refinement. Work the loop:

  1. Start with a benchmark set A table of question → expected query → expected answer. Use your four failing questions as the seed; grow it over time.
  2. Diagnose wrong answers Missing instruction? Vague? Ambiguous question? Inconsistent values (Tower Hamlets vs TowerHamlets)?
  3. Clarify agent instructions Fix source selection, response style, and terminology.
  4. Improve data source instructions Clarify filters, add typical value examples, enforce consistent terms.
  5. Add targeted examples Show correct joins, filter patterns, and output columns.
  6. Address join issues Document the LPI_Pseudonym relationship and add a join example. If joins get error-prone, consider flattening into one table.
MindsetTreat the agent as an evolving system that improves through regular testing and feedback - not a config you set once and forget.

Section 06

Power-ups

Two optional capabilities: deeper analysis, and control over how updates reach your agent.

Code Interpreter Preview

Gives your agent a secure, sandboxed Python environment to analyse the data it retrieves - calculations, correlations, combining sources, and visualisations. It generates and runs the Python for you; review the code and charts in the run steps.

Enable it in the UI

  1. Open your data agent
  2. Select the Tools tab
  3. Select "Add code interpreter", then Confirm.
First-time gotchaIn an existing chat, start a new chat or clear the thread before the agent picks up the new tool.

Questions to try on this data

  • Chart monthly DNA rate by directorate over 2025.
  • Is longer average contact duration correlated with fewer readmissions?
  • Forecast next quarter's admissions from the past two years.

Runtime

Every agent runs on a runtime - the orchestration, routing, and built-in query tools.

Standard GA

Default for new agents. Stable and predictable; updated infrequently. Choose it for production.

Preview

Latest tool and orchestration improvements before GA. Updated often; behaviour can shift. Choose it to evaluate upcoming changes.

ImportantRuntime controls core-component updates - not which sources you can add, and not the LLM. Model upgrades apply to both runtimes equally. A published agent keeps its runtime until you republish.

Surface your agent elsewhere

A published data agent isn't limited to the Fabric chat - it's a standalone artifact you can reach from other tools:

  • Microsoft 365 Copilot - surface natural-language insights inside M365 apps (Purview policies still apply to the underlying data).
  • Copilot Studio, Azure AI Foundry, and Teams - plug the agent into custom copilots and multi-agent workflows, where external orchestrators can invoke it.

However it's reached, the agent stays read-only and governed.

Manage changes (ALM)

  • Git integration - version-control your instructions, example queries, and data-source selections.
  • Deployment pipelines - promote an agent from development to test to production.
  • Diagnostics - monitor behaviour and troubleshoot query-generation and routing issues.

Section 07

Evaluate Preview

Prove your agent improved. Score it against a benchmark set with the SDK, in a notebook.

The workflow

  1. Install the SDK %pip install -U fabric-data-agent-sdk
  2. Load a ground-truth set Your failing questions plus their correct answers, as question / expected_answer pairs.
  3. Run the evaluation evaluate_data_agent compares the agent's answers to yours and stores the metrics.
  4. Review the results Pull a summary, then drill into per-question detail.
import pandas as pd
from fabric.dataagent.evaluation import (
    evaluate_data_agent, get_evaluation_summary)

df = pd.DataFrame(
    columns=["question", "expected_answer"],
    data=[
      ["How many patients were readmitted within 28 days?", "319"],
      ["How many DNAs are recorded in total?", "12687"],
    ])

eval_id = evaluate_data_agent(
    df, "MentalHealthAgent",
    table_name="hackathon_eval",
    data_agent_stage="production")

summary = get_evaluation_summary("hackathon_eval")

What you get back

  • Summary - totals of correct / incorrect / unclear answers, plus accuracy.
  • Details - per-question expected vs actual answer, a result flag, and a link to the evaluation thread.
  • Custom critic - supply your own critic_prompt (with {query}, {expected_answer}, {actual_answer}) for stricter or domain-specific judging.
Diagnostics buttonExport the agent's full configuration and execution steps - ideal for troubleshooting or working with Microsoft Support.

That's the loop

Build → instruct → add examples → tune routing → evaluate → repeat. Re-run your benchmark after each change and watch accuracy climb. Good luck - go build something brilliant.
